```python
@contextmanager
def get_db_ctx():
    conn = get_db()
    try:
        yield conn
        conn.commit()
    except Exception:
        conn.rollback()
        raise
    finally:

```

**Architecture requirements (per RESEARCH.md and CONTEXT.md decisions):**

1. **Pure ASGI class** — NOT BaseHTTPMiddleware. Uses `__init__(self, app)` and `async __call__(self, scope, receive, send)`.

2. **Single-pass enforcement** — All three rate limit categories (global, login, LLM) are checked in one `__call__` method:
   - Pre-request: check login limit if path == "/auth/login", check global limit for all paths, check LLM limit if path matches LLM_PATHS. If any check fails, send 429 and return.
   - Post-response (via wrapped send): increment global counter always, increment LLM counter if LLM path, increment login_failed counter if /auth/login returned 401.

3. **Fail-open on ALL DB operations** — Every `get_db_ctx()`, `conn.execute()`, and `conn.fetchone()` call MUST be wrapped in `try/except Exception`. On any exception: log the error via `logger.error(...)`, then allow the request to proceed (return True from check methods, silently return from increment methods).

4. **Login rate limiting specifics (per D-01, D-14):**
   - Track FAILED attempts only (HTTP 401 responses from /auth/login), not all login requests.
   - The pre-request check looks at the current login_failed count. If count >= 5 and window is still active (within 15 minutes), block with HTTP 429 and Chinese message: `{"detail": "登录尝试过多，请15分钟后再试"}`.
   - The post-response increment only fires when response_status == 401.
   - On successful login (200), do NOT increment the counter (successful logins should not count against the rate limit).

5. **LLM rate limiting (per D-03):**
   - Track by user_id, not IP. Extract user_id from Authorization header by calling `verify_token(token)`.
   - Match paths using `path.startswith(p)` against LLM_PATHS = frozenset({"/ai-generate-nodes", "/analyze-node", "/generate-question", "/generate-batch"}).
   - Pre-request: if user_id has >= 10 requests in the current 60-second window, block.
   - Post-response: increment LLM counter for that user_id.

6. **Sliding window algorithm (per D-13):**
   - For each (key, endpoint) pair, store (count, window_start) in rate_limits table.
   - On check: if window_start is older than the window duration, reset count to 0 (treat as new window).
   - On increment: if current window is still active, UPDATE count=count+1. If window expired, INSERT OR REPLACE with count=1 and new window_start.

7. **Periodic cleanup (per RESEARCH.md open question #2):**
   - Track a request counter (`self._request_count`). Every 100th request, run `DELETE FROM rate_limits WHERE window_start < datetime('now', '-15 minutes')`.
   - The cleanup is optional (fail-open — if it fails, just log and continue).

8. **IP extraction (per D-01):**
   - Extract IP from `scope.get("client")` — returns `(host, port)` tuple, use `host` as the IP.
   - If client is None, use "unknown" as a fallback IP.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client IP (scope["client"]) → middleware | Untrusted input — any IP can be spoofed via proxies |
| Authorization header → middleware | Untrusted input — token may be invalid, expired, or malicious |
| middleware → SQLite rate_limits table | Trusted boundary — middleware reads/writes rate limit counters |
| middleware → app | Rate limit decision gates whether request reaches endpoint handlers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02-01 | Spoofing | Login rate limiter | mitigate | Rate limiting by IP thwarts brute-force password guessing. 5 attempts per 15-min sliding window prevents burst attacks at window boundaries. |
| T-01-02-02 | Denial of Service | Rate limit DB operations | mitigate | ALL database calls wrapped in try/except Exception. On any DB error (OperationalError, disk full, WAL lock), the request proceeds. Logged at ERROR level for monitoring. This is fail-open — availability trumps rate limit enforcement during storage outages. |
| T-01-02-03 | Denial of Service | Global rate limiter | mitigate | 100 req/min per IP prevents resource exhaustion from high-frequency API calls. Separate LLM limit (10 req/min per user) prevents LLM API quota exhaustion. |
| T-01-02-04 | Spoofing | IP extraction from scope | accept | IP read from `scope["client"]` which uvicorn sets from the actual TCP connection. X-Forwarded-For header spoofing is a concern if behind a proxy without `--proxy-headers`, but the project's single-server deployment means the uvicorn process sees the real connecting IP (or Nginx's IP if proxied). Accept for now; if Nginx relay becomes active, configure `--proxy-headers` in uvicorn. |
| T-01-02-05 | Elevation of Privilege | LLM rate limit user extraction | mitigate | User ID extracted via `verify_token()` from auth.py. Invalid/expired tokens return None, and the rate limiter falls back to IP-based tracking for LLM endpoints. Valid tokens are cryptographically verified (HS256). |
| T-01-02-06 | Tampering | Rate limit counter overflow | accept | SQLite INTEGER is 64-bit signed (max 9.22e18). At 100 req/min, overflow would take ~1.75e14 years. Not a practical concern. |
</threat_model>
