Create `backend/db_migrate.py` with a single public function `run_migrations(conn, migrations_dir)`.

**Implementation requirements (per RESEARCH.md Pattern 3):**

1. **Schema version table creation:**
```sql
CREATE TABLE IF NOT EXISTS schema_version (
    version INTEGER PRIMARY KEY,
    applied_at TEXT NOT NULL
)
```

2. **Baseline detection:** Read `SELECT COALESCE(MAX(version), 0) FROM schema_version`. If result is 0, check if any user tables exist (query `sqlite_master` for tables other than `schema_version` and `sqlite_sequence`). If user tables exist, insert version=1 with current UTC ISO timestamp as the baseline. This handles existing installs where the schema already exists but no version tracking was in place.

3. **File discovery and ordering:** List files in migrations_dir ending with `.sql`, sort by integer prefix: `sorted(files, key=lambda f: int(f.split("_")[0]))`. Skip files where the version number is less than or equal to the current max version.

4. **Migration application:** For each unapplied migration file, read the SQL content, execute via `conn.executescript(sql)`, then `INSERT INTO schema_version (version, applied_at) VALUES (?, ?)`. Use `datetime.now(timezone.utc).isoformat()` for the timestamp.

5. **Transaction safety:** The entire function must be called within an existing transaction (i.e., conn is already inside a `get_db_ctx()` context manager from init_db()). Do NOT begin a new transaction inside run_migrations() — rely on the caller's transaction boundary.

6. **Error handling:** If `migrations_dir` doesn't exist, return silently (no migrations to apply). If a migration file can't be read or its SQL fails, let the exception propagate — the caller's transaction will rollback, preventing partially-applied migrations.

**Imports needed:**
```python
import os
import sqlite3
from datetime import datetime, timezone
```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | db_migrate.py | mitigate | Migration filenames parsed via `int(f.split("_")[0])` — malformed filenames raise `ValueError` and abort before any SQL executes. Migration SQL files are committed source files, not user-controlled input. |
| T-01-02 | Tampering | db_migrate.py | mitigate | All migration SQL uses `CREATE TABLE IF NOT EXISTS` to be idempotent against partial application. The caller's `get_db_ctx()` transaction wraps the entire migration run — any SQL failure rolls back the version INSERT too, preventing the "table exists but version not recorded" partial state. |
| T-01-03 | Denial of Service | database.py | accept | If migrations directory is missing or corrupted, init_db() proceeds with existing CREATE TABLE IF NOT EXISTS guarantees. The application still starts. This is fail-safe for the common case (missing migrations dir on first run before Task 1 completes). |
